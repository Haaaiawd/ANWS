# ADR-004: 多 AI IDE 单目标适配策略

**状态**: Accepted
**日期**: 2026-03-14
**关联需求**: [REQ-001] [REQ-003] [REQ-004]
**关联版本**: .anws v5

---

## 背景

v4 已确认 `anws` 不应长期维护多份独立模板副本，而应采用统一源 + 适配层的方向。但当时尚未拍板初始化时到底是“一次安装多个目标”还是“单次只安装一个目标”。随着 `ide_example/` 与现有 `.windsurf`、`.agents` 基线的对照，目录差异已经足够明确：如果继续允许一次安装多套目录，CLI 的行为边界、README 文案、managed files 与 update 路由都会显著复杂化。

---

## 决策

采用 **单目标 IDE 安装** 策略：

1. `anws init` 必须让用户选择一个目标 AI IDE。
2. CLI 只安装该目标所需的目录与文件。
3. `anws update` 默认只面向当前已安装目标进行升级。
4. 若未来需要切换目标，应作为显式迁移流程，而不是隐式发生在 `update` 中。

---

## 第一批目标

- Windsurf
- Antigravity / `.agents`
- Cursor
- Claude
- GitHub Copilot
- Codex

---

## 选择理由

- 把产品行为变得清晰：一次初始化服务一个环境。
- 降低更新边界复杂度：managed files 只需围绕一个目标求值。
- 让 README、CLI help、终端输出与用户心智一致。
- 避免“同一项目落多套 IDE 目录”带来的误提交、误更新和审查噪音。

---

## 后果

### 正面

- 初始化交互更明确
- manifest 可按目标动态生成
- update 更容易做到可预测与可验证
- 新增目标 IDE 时只需补充一套 adapter 规则

### 代价

- 需要增加目标选择交互
- 需要定义已安装目标检测逻辑
- 若用户确实要切换目标，需要额外迁移设计

---

## 约束

1. 不允许一次性安装多套目标目录作为默认行为
2. 目标差异不得直接散落在 `init.js` / `update.js` 的复制逻辑中
3. 终端输出必须展示当前目标 IDE 上下文
