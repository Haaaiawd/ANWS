# Sequential Thinking (顺序推理)

> "磨刀不误砍柴工" — 深度思考是高质量产出的前提

本技能提供**结构化的逐步推理框架**，帮助 AI 在复杂问题中保持清晰的思维链。

---

## 🎯 使命与定位

**这个技能是什么**: 提供逐步推理的思维框架，而非 MCP 工具调用。

**何时调用**:
- 问题需要多个相互关联的推理步骤
- 初始范围或方法不明确
- 需要在复杂性中找到核心问题
- 可能需要回溯或修正早期结论
- 想探索替代方案路径

**何时不调用**:
- 简单查询、直接事实、单步任务
- 问题已有明确的解决方案路径

---

## ⚠️ 核心原则

> [!IMPORTANT]
> **这是思维框架，不是 MCP 调用**
>
> 本 skill 是一个**纯思维框架**，AI 应在内部执行推理过程，而非调用外部工具。

## 🧭 推理流程 (Reasoning Process)

### Step 1: 初始思考 (Initial Thought)

**目标**: 建立推理起点，定义问题范围。

**思考模板**:
```markdown
**Thought 1/N**: [问题描述]

- **问题是什么？**: [清晰定义问题]
- **已知信息**: [列出已知条件]
- **未知信息**: [列出需要探索的部分]
- **初步假设**: [可能的解决方向]
```

**示例**:
```markdown
**Thought 1/5**: 优化数据库查询性能

- **问题是什么？**: API 响应时间 p95 > 500ms，需要优化
- **已知信息**: 使用 PostgreSQL，查询涉及 users 和 orders 表
- **未知信息**: 具体瓶颈在哪里？索引？N+1？锁竞争？
- **初步假设**: 可能是 N+1 查询问题或缺少索引
```

---

### Step 2: 逐步推理 (Sequential Reasoning)

**目标**: 逐步深入分析，每一步都有明确产出。

**推理规则**:
- 每一步只解决一个子问题
- 明确标注当前步骤编号和总预估步数
- 可以动态调整总步数

**思考模板**:
```markdown
**Thought {N}/{M}**: [当前推理主题]

- **分析**: [当前步骤的分析内容]
- **发现**: [从分析中得出的结论]
- **下一步**: [下一步要探索什么]
```

---

### Step 3: 修正与分支 (Revision & Branching)

**目标**: 当发现假设错误时，修正推理链或探索替代方案。

#### 修正 (Revision)

当新证据与早期结论矛盾时：

```markdown
**Thought 5/7** (修正 Thought 2):

- **原结论**: N+1 问题是瓶颈
- **新证据**: Profiling 显示瓶颈是缺少 join 列索引
- **修正**: 实际问题是索引缺失，而非 N+1
- **影响**: 需要重新评估优化方案
```

#### 分支 (Branching)

当存在多个可行方案时：

```markdown
**Thought 4/6** (分支 A: 缓存方案):
- **方案**: 使用 Redis 缓存热点数据
- **优点**: 减少数据库负载
- **缺点**: 需要处理缓存失效

**Thought 4/6** (分支 B: 索引方案):
- **方案**: 添加复合索引
- **优点**: 直接解决查询瓶颈
- **缺点**: 写入性能略有下降
```

---

### Step 4: 最终结论 (Final Conclusion)

**目标**: 总结推理过程，给出明确结论。

**思考模板**:
```markdown
**Thought {N}/{N}** (最终结论):

- **问题回顾**: [原始问题]
- **推理路径**: [关键推理步骤摘要]
- **结论**: [最终答案/决策]
- **行动建议**: [下一步具体行动]
```

---

## 📤 输出格式

**输出方式**: 在对话中直接输出推理过程，无需写入文件。

**格式示例**:
```markdown
## 深度推理: 优化数据库查询性能

**Thought 1/5**: 问题定义
- 问题: API p95 > 500ms
- 假设: 可能是 N+1 或索引问题

**Thought 2/5**: 分析查询模式
- 发现: users JOIN orders 查询频率最高
- 疑点: orders.user_id 列无索引

**Thought 3/5** (修正): 重新评估
- 原假设: N+1 问题
- 实际: EXPLAIN 显示全表扫描
- 结论: 缺少索引是主因

**Thought 4/5**: 方案对比
- 方案 A: 添加索引 → 简单有效
- 方案 B: 缓存 → 复杂度高
- 选择: 方案 A

**Thought 5/5** (最终结论):
- 根因: orders.user_id 缺少索引
- 解决: CREATE INDEX idx_orders_user_id ON orders(user_id)
- 预期效果: p95 < 100ms
```

## 🛡️ 最佳实践

1. **表达不确定性**: 使用"可能"、"需要验证"、"不确定"等词汇
2. **展示推理过程**: 不仅给出结论，还要说明如何得出
3. **自由修正**: 修正思路是正常的，不要害怕推翻早期假设
4. **果断分支**: 当存在多个可行方案时，明确探索每个方案
5. **动态调整**: 不要锁定初始预估步数，根据实际情况调整
6. **保持清晰**: 每一步都应足够独立，可单独理解
